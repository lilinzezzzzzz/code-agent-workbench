# AI 大模型交流最佳实践

本文提供一组可以直接追加在消息末尾的短句，用于在已经给出背景、想法、
方案或内容后，引导 AI 给出更明确、更有批判性和更可执行的回答。

例如：

```text
我设计：xxxxx。

请判断这样设计是否合理。不要默认我的思路正确，先给出明确结论，再指出
最关键的问题和可执行的改进建议。
```

## 一、核心原则

“这样处理合适吗”“这样合理吗”这类问法本身没有错，但容易让 AI 给出礼貌、
宽泛的回答。通常只需要在原问题后增加一两个限制：

- **要求结论**：先明确回答合理、不合理，还是需要修改。
- **要求批判**：不要默认我的判断正确，不要为了附和而弱化问题。
- **要求重点**：只指出最重要的 3 个问题，并按优先级排序。
- **要求行动**：给出具体修改建议，不要只描述问题。
- **要求边界**：信息不足时明确指出，不要自行补充事实。
- **要求取舍**：建议替代方案时，同时说明收益、代价和风险。

不需要一次把这些要求全部加上。根据当前对话，选择最重要的两到三项即可。

## 二、通用追加句

### 最短版本

```text
请直接指出其中最关键的问题，并给出具体改进建议。
```

### 推荐版本

```text
不要默认我的判断正确。请先给出明确结论，再指出最关键的问题和可执行的
改进建议。
```

### 严谨版本

```text
请不要为了附和我而弱化批评。先给出明确结论，再按优先级指出问题、依据
和改进建议；无法根据现有信息判断的部分请明确说明。
```

## 三、常用问法优化

### 1. “这样处理合适吗？”

原问法：

```text
我准备这样处理：xxxxx。这样处理合适吗？
```

推荐：

```text
我准备这样处理：xxxxx。这样处理合适吗？请先给出明确结论，再指出最值得保留和最需要修改的部分。
```

偏重问题：

```text
这样处理合适吗？不要只做正面评价，请重点指出可能导致结果不理想的地方。
```

偏重行动：

```text
这样处理合适吗？如果需要改进，请直接告诉我应该改什么、为什么，以及如何改。
```

### 2. “这样设计合理吗？”

原问法：

```text
我设计：xxxxx。我这样设计合理吗？
```

推荐：

```text
我设计：xxxxx。请判断这样设计是否合理。不要默认我的思路正确，先给出
明确结论，再指出最关键的问题和改进建议。
```

偏重简洁性：

```text
我这样设计合理吗？请特别检查是否存在不必要的复杂度，以及能否用更简单
的方式达到同样目标。
```

偏重风险：

```text
我这样设计合理吗？请重点检查边界情况、失败路径和后续维护风险。
```

偏重替代方案：

```text
我这样设计合理吗？如果不合理，请给出一个更合适的替代方案，并说明它的
收益和代价。
```

### 3. “你有什么更好的建议或方案？”

原问法：

```text
当前方案是：xxxxx。你有什么更好的建议或方案？
```

推荐：

```text
你有什么更好的建议或方案？请优先给出能够实际落地、且收益明显高于实施
成本的建议。
```

需要强推荐：

```text
如果有更好的方案，请不要只罗列选项。比较收益、成本和风险后，明确推荐
一个你认为最合适的方案。
```

限制范围：

```text
你有什么更好的建议？请围绕当前目标改进，不要引入无关功能或不必要的
大规模重构。
```

### 4. “会有什么问题和漏洞？”

原问法：

```text
我的方案是：xxxxx。会有什么问题和漏洞？
```

推荐：

```text
这个方案可能有什么问题和漏洞？请按严重程度排序，并说明触发条件、影响
和对应的修复建议。
```

偏重失败分析：

```text
请从失败路径出发检查这个方案，重点考虑边界情况、异常流程、外部依赖失败
和用户误操作。
```

偏重安全：

```text
请从攻击者视角检查可能的安全漏洞、权限绕过、数据泄露和滥用方式，并区分
现实风险与纯理论风险。
```

避免无边界发散：

```text
请只列出最值得处理的 5 个风险，按影响和发生概率综合排序，不要用低概率
问题稀释重点。
```

### 5. “哪里需要精简优化？”

原问法：

```text
当前内容是：xxxxx。哪里需要精简优化？
```

推荐：

```text
哪里需要精简优化？请指出可以删除、合并或改写的部分，并给出优化后的
版本，不要改变核心目标。
```

偏重内容表达：

```text
请删除重复和低价值内容，保留关键结论与必要依据，然后直接给出精简后的
完整版本。
```

防止过度精简：

```text
请在不损失准确性、必要背景和关键约束的前提下精简，不要单纯追求篇幅更短。
```

### 6. “根据我给出的内容补充或者优化”

原问法：

```text
根据我给出的内容进行补充或者优化。
```

推荐：

```text
请根据我给出的内容补充和优化，保留原意并直接给出修改后的完整版本；不要
虚构事实，不确定的内容请单独标注。
```

允许重写结构：

```text
可以重新组织结构和表达，但必须保留核心意图。请先给出优化后的完整版本，
再简要说明主要修改。
```

只允许小幅修改：

```text
请只修正表达、结构和明显缺漏，不要扩大主题，也不要加入未经我确认的新观点。
```

需要主动补漏：

```text
请在优化表达的同时检查逻辑缺口。必要的补充可以加入，但请将新增内容和
原有内容明确区分。
```

## 四、按目标选择追加句

### 希望 AI 不要一味附和

```text
不要默认我的判断正确。如果前提、推理或结论有问题，请直接指出。
```

```text
请站在反方立场提出最有力的反对意见，再判断这些意见是否足以推翻当前方案。
```

### 希望 AI 先给结论

```text
请先用一句话给出明确结论，再展开原因。
```

```text
请明确回答：可以直接采用、修改后采用，还是不建议采用。
```

### 希望 AI 抓住重点

```text
只指出最重要的 3 个问题，并按优先级排序。
```

```text
如果只能修改一处，最应该修改哪里？请说明原因。
```

### 希望建议可执行

```text
不要只描述问题。每个问题都要给出具体修改动作和预期效果。
```

```text
避免使用“进一步优化”“加强管理”这类模糊表述，请告诉我具体应该怎么做。
```

### 希望比较方案

```text
请比较当前方案和替代方案的收益、成本、风险与适用条件，并明确推荐一个。
```

```text
如果现有方案已经足够好，请直接说明，不必为了提供建议而强行设计新方案。
```

### 希望控制回答长度

```text
请先简要回答，控制在 5 点以内；只有关键问题需要展开说明。
```

```text
先给摘要，再展开最重要的 3 个问题，次要问题只列标题。
```

### 希望降低幻觉

```text
请区分已知事实、合理推断和无法确认的信息，不要自行补充未经验证的事实。
```

```text
如果现有信息不足以判断，请明确指出缺少什么信息，不要猜测。
```

### 希望结论可验证

```text
请说明判断依据，以及可以通过什么测试、数据或现象验证你的结论。
```

```text
请为改进建议补充可观察的验收标准。
```

## 五、推荐的组合方式

一次追加两到三个要求通常足够。推荐从以下四类中各选一句：

1. **结论**：先给出明确结论。
2. **批判**：不要默认我的判断正确。
3. **重点**：只指出最关键的 3 个问题。
4. **行动**：给出具体修改建议。

### 通用评审组合

```text
请先给出明确结论，不要默认我的判断正确。只指出最关键的 3 个问题，并给出
具体改进建议。
```

### 设计评审组合

```text
请判断这个设计能否直接采用、需要修改，还是不建议采用。重点检查不必要的
复杂度、边界情况和维护风险；如果有更好的方案，请说明收益和代价。
```

### 风险审查组合

```text
请从失败路径出发，列出最重要的 5 个风险，按严重程度排序，并说明触发条件、
影响和修复建议。
```

### 内容优化组合

```text
请保留原意，删除重复和低价值内容，补齐必要的逻辑缺口，然后直接给出优化后
的完整版本。新增或不确定的内容请单独标注。
```

## 六、使用建议

- 上下文已经充分时，直接追加短句，不要重复描述背景。
- AI 回答过于正面时，追加“不要默认我的判断正确”。
- AI 回答过于宽泛时，限制问题数量并要求按优先级排序。
- AI 只指出问题时，追加“给出具体修改动作和预期效果”。
- AI 罗列太多方案时，要求比较后明确推荐一个。
- 涉及事实准确性时，要求区分事实、推断和未知信息。
- 简单问题使用一句追加语即可，不需要为了形式完整而堆叠所有要求。

## 七、默认推荐句

如果不确定应该追加什么，可以直接使用：

```text
不要默认我的判断正确。请先给出明确结论，再指出最关键的问题和可执行的
改进建议；如果信息不足，请说明哪些部分无法判断。
```
