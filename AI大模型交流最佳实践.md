# AI 大模型交流最佳实践

本文整理一组可以直接追加在消息末尾的短句。适用于你已经给出背景、想法、
方案或内容后，要求 AI 给出更明确、更有批判性、更可执行的回答。

示例：

```text
我设计：xxxxx。

请判断这个设计能否直接采用、需要修改，还是不建议采用。不要默认我的思路
正确，先给出明确结论，再指出最关键的问题和改进建议。
```

## 1. 核心原则

普通问法如“这样合理吗”“哪里需要优化”本身没有错，但容易得到礼貌、宽泛
的回答。更有效的做法是在原问题后追加 2 到 3 个约束：

- **目标**：先围绕当前目标判断，不脱离目标做泛泛优化。
- **结论**：先明确回答可采用、需修改，还是不建议采用。
- **批判**：不要默认我的判断正确，有问题直接指出。
- **重点**：只列最重要的问题，并按优先级排序。
- **行动**：给出具体修改动作，不只描述问题。
- **验证**：给出判断建议是否有效的验收标准、测试方式或观察指标。
- **边界**：信息不足时明确说明，不自行补充事实。
- **取舍**：建议替代方案时说明收益、成本和风险。
- **标准**：判断是否符合最佳实践时，说明参考的原则、适用条件和例外情况。

不需要一次堆叠所有要求。简单问题用一句追加语即可；复杂问题优先选择
“目标、结论、批判、重点、行动”这几类约束；需要落地验收时再补充“验证”。

## 2. 默认可复制句

### 最短版

```text
请直接指出其中最关键的问题，并给出具体改进建议。
```

### 推荐版

```text
不要默认我的判断正确。请先给出明确结论，再指出最关键的问题和可执行的
改进建议；如果信息不足，请说明哪些部分无法判断。
```

### 严谨版

```text
请不要为了附和我而弱化批评。先给出明确结论，再按优先级指出问题、依据
和改进建议；无法根据现有信息判断的部分请明确说明。
```

## 3. 常见场景

### 方案 / 设计评审

```text
请评审这个方案是否适合采用。先给出明确结论，再按优先级指出关键问题、依据和改进建议；
```

```text
请重点检查这个设计是否存在不必要的复杂度、边界情况、失败路径和后续维护
风险；如果有更简单的方案，请说明收益和代价。
```

### 最佳实践符合性检查

```text
请判断这个做法是否符合当前领域的最佳实践。先给出明确结论，再说明它符合
或不符合哪些原则、适用条件和例外情况。
```

```text
如果这不是最佳实践，请说明问题严重程度、现实影响、是否必须修改，以及最小可行的改法。不要只给抽象原则。
```

### 风险 / 漏洞检查

```text
请从失败路径出发检查这个方案，列出最重要的 5 个风险，按严重程度排序，
并说明触发条件、影响和修复建议。
```

```text
请从攻击者视角检查可能的安全漏洞、权限绕过、数据泄露和滥用方式，并区分
现实风险与纯理论风险。
```

### 替代方案 / 取舍

```text
如果有更好的方案，请不要只罗列选项。请比较当前方案和替代方案的收益、
成本、风险与适用条件，并明确推荐一个。
```

```text
如果现有方案已经足够好，请直接说明，不必为了提供建议而强行设计新方案。
请围绕当前目标改进，不要引入无关功能或不必要的大规模重构。
```

### 内容优化 / 重写

```text
请指出可以删除、合并或改写的部分，并给出优化后的版本；不要改变核心目标。
```

```text
请保留原意，删除重复和低价值内容，补齐必要的逻辑缺口，然后直接给出优化后
的完整版本。新增或不确定的内容请单独标注。
```

```text
请在不损失准确性、必要背景和关键约束的前提下精简，不要单纯追求篇幅更短。
```

## 4. 可选约束

### 控制长度

```text
请先简要回答，控制在 5 点以内；只有关键问题需要展开说明。
```

### 抓住重点

```text
只指出最重要的 3 个问题，并按优先级排序。如果只能修改一处，请说明最应该
修改哪里以及原因。
```

### 要求可执行

```text
不要只描述问题。每个问题都要给出具体修改动作和预期效果，避免使用
“进一步优化”“加强管理”这类模糊表述。
```

### 降低幻觉

```text
请区分已知事实、合理推断和无法确认的信息。不要自行补充未经验证的事实；
如果信息不足，请明确指出缺少什么。
```

### 要求验证标准

```text
请说明判断依据，并为改进建议补充可观察的验收标准。
```
